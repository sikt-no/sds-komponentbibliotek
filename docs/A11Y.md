# Accessibility

All components should be accessible!  
More on [accessibility in Sikt](https://isikt.sharepoint.com/sites/Innsikt-OmOss/SitePages/Universell-utforming.aspx) and [accessibilty for developers in Sikt](https://platon.sikt.no/uu/).

## Testing

Code is automatically checked with [eslint-plugin-jsx-a11y](https://github.com/jsx-eslint/eslint-plugin-jsx-a11y). All components should be tested for accessibility issues with [jest-axe](https://github.com/nickcolley/jest-axe).

**Note #1** These tests only catch a certain percentage of issues and manual testing is still required by both designers & developers.

**Note #2** Even though the components have unit tests you still need to write integration tests in your own application where you have used them.

## Relative sizes

Relative size tokens (`--sds-size-text-<size>`) should be used on user font size setting scalable properties like `font-size`, `line-height`, etc for accessibility purposes.

## Motion

The Core package CSS reset includes a media query for reduced motion that disables animations and transformations.

## Elements

### `<abbr>`

The [abbreviation element is not accessible by itself](https://developer.mozilla.org/en-US/docs/Web/HTML/Element/abbr#accessibility_concerns). It can be hard to understand without correct text content and can't be accessed by keyboard. One way to solve this is to simply spell out the acronym or abbreviation in full the first time it is used on a page, instead of using the `<abbr>`-element.

Another alternative is to use `<abbr>` with accessible markup. `tabindex` makes is accessible by keyboard and the `sds-typography-abbr` class name will display the content.

```html
<abbr class="sds-typography-abbr" tabindex="0" title="Sikt Design System">
  SDS
</abbr>
```

There is an `<Abbrevation />` component in the core package that does this for you.

### `<input type="date">`

The input element type date is not accessible and building one that is can be hard. Give the user an alternative to selecting a date in a calendar, like typing into input fields.

### `<... disabled>`

Disabled state are bad for all users, trying to figure out why and how to shift state to enabled. Instead of disabling interactive elements have them enabled and give the user feedback on what action needs to be taken. For example with error messages about filling out a input before going to the next step.

### Maps

Interactive maps such as Google Maps are not accessible. However these are not required to be as long as the information is conveyed in an alternative form, like an address, for more information see [UUTilsynet](https://www.uutilsynet.no/regelverk/kva-seier-forskrifta/153#unntak_for_kart_og_karttenester_).

## Stylesheets

### CSS content

If the CSS content property carries meaning it needs an accessible alternative [since it is not accessible by itself](https://developer.mozilla.org/en-US/docs/Web/CSS/content#accessibility_concerns). CSS alternatives are not fully supported yet, see [CSS property: content: Alternative text](https://caniuse.com/mdn-css_properties_content_alt_text) or [CSS property: alt](https://caniuse.com/mdn-css_properties_alt). So for the time being valid options are using `aria-label` or a `.sds-screen-reader-only` element containing the additional information.
